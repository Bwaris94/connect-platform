is that it?

of course not. i mean, it is not just about sync and async code. doing backend code is becoming more and more stale and repetitive and boring by the day, while it should essentially be focusing on the core business logic that you want to do with your service.

we believe the main reason is that software development is essentially two separate processes mashed into each other in an unholy manner:

micrologic, i.e. creating building blocks

this is what programming languages were created for. to create some building blocks, which are usually more independent from the business logic and hence highly reusable. and,

macrologic, or building something with those building blocks

which is the larger-scope logic of the product. think user journey, process analysis, or however context a product designer/manager/owner/whatever might like to describe the product. this is far less reusable as it is highly entangled in the product definition and goals itself.

these two processes require different skill sets, technically skilled people excell at doing the former while you require product-manager-ish people to correctly do the latter.

they also require different mind-sets as well, while creating some building blocks you are not thinking of user and product's flow, you are thinking performance, reusability, etc, yet while you are connecting them together to form a backend service as part of your product's business logic you are not thinking of performance or reusability, you are thinking of what happens to your users and where the data flows in your system.

they even require different tools, since the building blocks themselves are usually better off done in classic synchronous codes, while the high-level logic of a web service is definitely asynchronous in nature, and as discussed above, programming languages aren't the best tools to do the latter.

however, in the end of the day, we are now doing them all in one place through programming them. product people aren't as hands-on (as they cannot be with complex codes we have), so although they offer some idea of what is to be implemented, the work is on the shoulder of technical people still. in this light, it is rather obvious why separating these two with a tool like CONNECT platform and putting people with the correct skills and mindsets in charge of each greatly increases speed and productivity.

this is a really early stage project. it has yet to be get more and more integrations and feature rich, and it has yet to be truly tested in heavy-load environments. however, if you are interested, you can give it a try: 


npm install connect-platform --save

and/or better yet, you can clone the code and contribute to this project and help reduce the pain of next backend project someone like you and me would need to do: 


# clone from git
git clone https://github.com/loreanvictor/connect-platform.git

# go to the directory
cd connect
cd platform

# install dependencies
npm install

# go to panel directory and install its dependencies
cd panel
npm install

# build the panel
npm run angular-build

# go back and start the project
cd ..
npm start
