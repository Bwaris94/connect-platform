I hate it that building API micro-services isn't as easy as just a few clicks and drags, specifically when most of the time you don't need extremely convoluted logical or hefty computational code on them. the goal of this project is to change that.

hmmm ...

cool, but why?

the intuition is to seperate micro-logic and macro-logic, i.e. logic that has a limited domain (like a password check, database query, etc) and logic whose scope encapsulates the whole service, as micro-logic is extremely repetetive and when done in the correct structure heavily reusable, and it is the macro-logic of each service that is unique and requires work.

however, without that seperation, the work that needs to be done gets cluttered up and I believe that is one of the main reasons that with all the available tools we have today it is still a cumbersome thing to do.

I think by seperating the two and encapsulating micro-logic into reusable chunks, the main task that remains, which is defining the macro-logic, will be an extremely easy one, easy enough to be able to be done via a visual panel as the one demonstrated above.

and since that is the main design intuition here, I hope that with the tools provided via this project even pure coding (of nodejs api micro-services) becomes far more streamlined and structured as well.
